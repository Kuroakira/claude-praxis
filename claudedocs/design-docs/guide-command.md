# /guide コマンド

## Overview

コードベースの「理解のためのガイドドキュメント」を生成するコマンド。構造分析（`/analyze`）がAIと人間の構造的判断材料を提供し、Design Docが設計決定の「なぜ」を記録する一方で、「コードを読みながら仕組みを理解するための手引き」は存在しない。`/guide`はこのギャップを埋める — ステップバイステップのナラティブウォークスルーで、読者がコードと並行して理解を深められるドキュメントを生成する。

## Context and Scope

### 3種類のドキュメントが異なる問いに答える

Claude Praxisでは開発の各局面で異なるドキュメントが生成されるが、それぞれが答える問いは異なる:

| ドキュメント | 答える問い | 読者 | 詳細度 |
|-------------|-----------|------|--------|
| `/analyze` レポート | この構造はどうなっているか？摩擦はどこか？ | AI + 人間（判断材料として） | 構造と依存関係を俯瞰 |
| Design Doc | なぜこの設計を選んだか？ | チーム（意思決定の記録として） | WHY重視、HOW抑制 |
| `/guide` ガイド | このコードはどう動くか？どう読めばいいか？ | 人間（学習・理解のために） | HOWを丁寧に、比喩も活用 |

`/analyze`と`/guide`は特に混同しやすい。両方とも「コードベースを理解する」ための文書だが、目的が根本的に異なる。`/analyze`は「構造的にどうなっているか」を客観的に記述する判断材料であり、AIが設計判断のインプットとして消費する。`/guide`は「この仕組みをどう理解すればいいか」を主観的に案内する学習支援であり、人間がコードを読みながら消費する。

### 既存ツールの限界

CodeTourやSwimmのようなウォークスルーツールはこのギャップを認識しているが、IDE拡張として実装されており、Claude CodeのMarkdownベースのエコシステムとは合わない。テキストベースで完結し、どの環境でも読めるMarkdownガイドが必要。

### ガイドが解決する具体的な場面

- 新しいチームメンバーがコードベースにオンボーディングする時
- 他チームのモジュールを初めて読む時
- 数ヶ月ぶりに自分が書いたコードに戻る時
- コードレビューのために背景理解が必要な時

いずれの場面でも、「構造図を見せられても理解できない。コードの横で、順を追って説明してほしい」というニーズがある。

## Goals / Non-Goals

### Goals

- **ナラティブガイドの生成**: コードを並行して読みながら理解を深められる、ステップバイステップのウォークスルードキュメント
- **柔軟なスコープ**: モジュール、ディレクトリ、横断的関心事、プロジェクト全体
- **全体像を見失わない構成**: 全体像 → フォーカス → 全体像に戻る → 次のフォーカス のリズムで、読者は詳細に入っても常に「今どこにいるか」がわかる
- **検証可能な説明**: すべてのコード解説はファイルパスとシンボル名を参照し、読者がコード上で即座に確認できる

### Non-Goals

- **IDE連携**: CodeTourのようなインラインアノテーション。Claude Codeのエコシステムに合わず、IDE依存が生まれる
- **自動更新**: コード変更に連動したガイドの自動同期。上書きモデル（再実行で更新）で対応する
- **読者レベルパラメータ**: 初心者/上級者の切り替え。単一のナラティブスタイルで、概念導入から始めて段階的に詳細化する構成自体が、幅広い読者に対応する。再検討条件: 実際の使用で、経験豊富なエンジニアが導入部分を冗長と感じるフィードバックが繰り返し出た場合

## Proposal

### ドキュメント構造: 全体像起点のズームイン/ズームアウト

ガイドの構造は「全体像 → フォーカス → 全体像に戻る → 次のフォーカス」というナビゲーションパターンに従う。

この構造が必要な理由は、詳細に入ると全体像を見失うという人間の認知特性にある。線形に詳細を積み上げるだけでは、読者は「今読んでいる部分が全体のどこに位置するのか」を見失い、やがてドキュメントを読むのをやめる。読まれないガイドは存在しないのと同じである。

具体的なパターン:

1. **全体像を先に提示**: 比喩や例えを用いて、システムの全体像を読者の頭に描く。詳細コードに入る前に「これは何をするシステムか」を直感的に理解できる素地を作る
2. **フォーカス**: 全体像の中から1つの領域を選び、詳細に案内する。エントリーポイントからデータフロー方向に沿って選択することで、案内順序を再現可能にする
3. **全体像に戻る**: 詳細を終えたら必ず全体像に立ち返り、「今読んだ部分は全体のここにあたる」と位置づけを確認する
4. **次のフォーカスへ**: 全体像の中から次の領域を選び、再び詳細に入る

このパターンの核心は「帰還」にある。読者が細部に入った後、全体像のどこにいるかを常に確認できる構造にすることで、理解が断片化しない。具体的なセクション構成はスキル内で定義するが、「フォーカスの後に全体像へ戻る」というリズムは設計原則として固定する。

### 正確性保証: Drill-to-Source原則

`/analyze`レポートには「Iron Law: すべての所見は具体的なファイルパスとシンボルを参照する」という原則がある。ガイドにも同じ原則を適用する。

ガイドは仮説文書である。AIが生成した説明がすべて正確である保証はない。しかし、すべての説明がファイルパスとシンボル名で裏付けられていれば、読者は「この説明は正しいか？」をコードを開いて即座に検証できる。検証不能な曖昧な説明（「この関数は〜を処理する」）は禁止し、検証可能な具体的参照（「`src/auth.ts`の`validateToken`関数は〜を処理する」）を強制する。

この原則により、たとえAIの解説に誤りがあっても、読者はそれを発見できる。「信頼して読む」ではなく「検証しながら読む」文書にすることで、誤った理解の定着を防ぐ。

### 生成戦略: Explore-then-Write

ガイド生成は2フェーズで実行する:

1. **Explore**: Scoutエージェントがコードベースを探索し、ファイル構造、主要シンボル、依存関係、データフローを把握する
2. **Write**: メインエージェントがScoutの探索結果を入力として受け取り、ナラティブを一貫して執筆する

この分離が重要な理由は語り口の一貫性にある。複数のエージェントが断片的にガイドを書くと、文体、詳細度、比喩の使い方がセクションごとに変わり、読み物として破綻する。単一の語り手（メインエージェント）が全体を通して案内することで、読者は一人のメンターに教わるような一貫した体験を得る。

### スコープ制御: 段階的深度制限

`/analyze`と同様、概要スキャン → 重要エリアの深掘りという段階的アプローチでスコープを制御する。探索エージェントの数に上限を設け、1回のガイド生成で扱える深度を制限する。具体的な上限値はスキルで定義する。

ガイドは`/analyze`よりも出力量が多い傾向にある。`/analyze`は構造を俯瞰する幅優先の文書だが、ガイドは理解を積み上げる深さ優先の文書であり、同じスコープでも文章量が大きくなる。このため、「プロジェクト全体」スコープでは概要レベルのガイド（全体像と主要コンポーネントの役割）に留め、個別モジュールの詳細ウォークスルーは`/guide [module]`で分割する。

### /analyzeとの関係: 完全独立

`/guide`は`/analyze`に依存しない。Scoutエージェントが毎回コードベースを直接探索する。既存の分析結果を明示的に参照・検出する仕組みは設けない。

ユーザーが`/analyze`を先に実行していた場合、その知識はユーザー自身のコンテキストとして活用される（「このモジュールの hooks を`/guide`して」のように具体的なスコープ指定につながる）。しかしコマンド間の暗黙的なデータ結合は、条件分岐と品質差を生むため避ける。

### レビューモデル: Light（document-quality + architecture）

ガイドのリスクは2つある: (1) 構造の読みにくさ・用語の不一致（`document-quality`がカバー）、(2) ナラティブがシステムの実際の境界を正しく反映しているか（`architecture`がカバー）。`document-quality`は`rules/document-quality.md`の基本原則のみ適用する（`rules/design-doc-format.md`はDesign Doc専用であり、ガイドには適用しない）。

内容の正確性についてはDrill-to-Source原則がファイル参照を強制するため、読者自身が検証可能な形になっている。ガイドは仮説文書として位置づけ、「正しさの保証」ではなく「検証可能性の保証」を設計方針とする。

### フォーマット定義: Skill内部

ガイドドキュメント専用のフォーマットをスキルのSKILL.md内で定義する。Design Doc format（WHY重視・HOW抑制）ともanalysisレポート（構造記述・摩擦検出）とも目的が異なるため、独立した構成を持つ。ただし、document-quality rulesの基本原則（用語一貫性、段階的詳細化、セクション自己完結性）は共通で適用される。

## Alternatives Considered

### A: /analyzeの拡張としてガイド機能を追加

`/analyze`に「ガイドモード」フラグを追加し、レポート生成時にナラティブ形式を選択できるようにする案。

却下理由: `/analyze`と`/guide`は答える問いが異なる。`/analyze`は「構造がどうなっているか」を客観的に記述する判断材料、`/guide`は「どう理解すればいいか」を案内する学習支援。同一コマンドに両方を持たせると、レポートの書き方やScoutへの指示が目的によって分岐し、コマンド内部の複雑度が上がる。SRP（単一責任原則）に反する。

再検討条件: `/analyze`レポートの読者とガイドの読者が完全に同一（AIも人間も同じ文書を同じ目的で消費する）になった場合。現時点では読者と消費パターンが明確に異なる。

### B: Design Doc形式でガイドを書く

既存のDesign Doc formatをガイドに流用する案。フォーマット定義の手間が省ける。

却下理由: Design Docの設計思想は「WHY重視・HOW抑制」— コードの実装詳細は書かず、設計判断の理由を記録する。ガイドは正反対で、HOW（コードの仕組み）を丁寧に、比喩を交えて解説する必要がある。Design Doc formatのルール（「コード例はデフォルトで含めない」「ディレクトリ構造は書かない」）がガイドの目的と矛盾する。

再検討条件: Design Doc formatが「目的別テンプレート」をサポートし、WHY/HOWバランスをテンプレートごとに変更できるようになった場合。

### C: CodeTour/Swimm的なIDE統合形式

VS Code拡張やSwimmのようなインラインアノテーション形式。コードの行に直接コメントを埋め込み、IDE上でステップバイステップ案内する。

却下理由: Claude CodeのエコシステムはMarkdownベースであり、IDE拡張への依存を避けている。CodeTour形式はJSON定義ファイルが必要でIDEなしでは読めず、Swimm形式は専用SaaS依存。テキストベースで完結し、どの環境でも読めるMarkdownがこのプロジェクトに最も適合する。

再検討条件: Claude Codeが公式にIDEアノテーション機能（コードの横にMarkdownを表示する仕組み）を提供した場合。

## Cross-Cutting Concerns

### コンテキストウィンドウへの影響

ガイド生成はコードベース探索を伴うため、トークン消費が他のコマンドより大きくなる可能性がある。3つの軽減策を組み合わせる:

1. **Explore-then-Write分離**: Scoutエージェントを別プロセスとして実行し、探索結果のみをメインコンテキストに返す
2. **段階的深度制限**: 探索エージェント数に上限を設けて探索範囲を制限
3. **スコープ委譲**: プロジェクト全体スコープは概要レベルに留め、深掘りは`/guide [module]`に分割

## Concerns

### ドキュメントの陳腐化（/analyzeより高リスク）

コード変更に伴いガイドが古くなるリスクは、`/analyze`レポートよりも高い。`/analyze`レポートは主にAIが`/design`や`/implement`のワークフロー中に再実行して消費する — 常に最新版が生成される。一方、ガイドは人間がブックマークし、チームで共有し、数ヶ月後に参照する。この消費パターンの違いにより、陳腐化したガイドが長期間使われ続けるリスクがある。

対策:

- **上書きモデル**: 同一スコープで`/guide`を再実行すると最新版で上書き。再生成のコストは低い
- **生成日時の明記**: ガイドヘッダーにタイムスタンプを含め、読者が鮮度を判断可能にする
- **Drill-to-Source参照**: 各ステップにファイルパスとシンボル参照を含めることで、コード側が変更されていれば読者がそれに気づける構造にする

自動的な陳腐化検出（「このガイドは古くなっています」通知）は含まない。この判断は、ガイドの利用頻度と再生成コストのバランスを実際の使用で確認してから行う。

### 探索の不完全性

Scoutの探索がスコープ全体をカバーできない場合がある。エントリーポイントが不明確なコードベース（ライブラリ等）、探索ターン数の制限による打ち切り、複雑な間接依存の見落しなどが原因となる。`/analyze`レポートにはConfidence Boundary（評価済み/未評価の明示）があり、ガイドにも同等の仕組みが必要。ガイドの末尾に「カバー範囲」セクションを設け、探索でカバーした範囲とカバーできなかった範囲を明記する。

### 生成品質のばらつき

コードベースの複雑さ、スコープの広さ、コードの読みやすさによって、生成されるガイドの品質は変動する。`document-quality`レビューが構造品質（用語一貫性、段階的詳細化、セクション自己完結性）を保証し、Drill-to-Source原則がすべての主張に検証可能な参照を強制する。

ただし、説明の「正しさ」自体はAIの理解精度に依存する。ガイドは仮説文書として位置づけ、読者に「検証しながら読む」ことを促す。これは`/analyze`レポートと同じスタンスであり、AI生成ドキュメントの限界を正直に認めた上で、検証可能性を設計に組み込むアプローチ。

## Review Checklist

- [ ] アーキテクチャ: 既存のcommand/skill構造に自然にフィットするか
- [ ] ドキュメント品質: ガイドフォーマットがdocument-quality rulesの基本原則と整合するか
- [ ] スコープ: `/analyze`との責務境界が明確で、両コマンドが独立して価値を提供するか
- [ ] パフォーマンス: コンテキストウィンドウ圧迫の軽減策が十分か
- [ ] 正確性: Drill-to-Source原則が仮説文書としての検証可能性を確保しているか
